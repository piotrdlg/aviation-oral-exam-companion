# HeyDPE — UI / UX / Art Direction Audit

> Phase 6B · W6B.1 · 2026-06-11 · Method: a 6-lens multi-agent audit over **20 real screenshots** (every public + authenticated screen, plus the live exam, at desktop 1440×900 and mobile 390×844) and the source. **71 findings** (6 blocker · 16 high · 33 medium · 16 low). Companion: `2026-06-11-redesign-spec.md`, mockups under `docs/design/mockups/`.

## Verdict

HeyDPE has a genuinely distinctive point of view and the bones of a premium product — the cockpit/avionics direction, the SaaS landing structure, and clean modals like Discard show real craft — but right now it sits at solid-indie/competent-dark-SaaS, well short of an Awwwards or Apple-bar standard. The gap is dominated by a few credibility-killers on the screens that matter most: the core voice-exam renders as a broken-looking empty void, the Progress dashboard reports zeros next to real activity, and a cluster of WCAG contrast/target/focus failures undercut a product that explicitly advertises accessibility. Fix those blockers and impose a single typographic and glow/contrast system (the tokens are already there), and this jumps from 'looks unfinished in places' to 'distinctive and trustworthy' — the award-level distance is then mostly disciplined consistency, not a redesign.

## The blockers — fix these or the rest doesn't matter

1. Live exam chat renders as a ~75% empty void with the single examiner message stranded at the top and the input bar floating far below — the core paid screen looks broken/half-loaded. Constrain the message area (give it a max-height and bottom-anchor/justify content so the latest message sits near the input), cap the bubble to ~65ch, and add a subtle 'new message' anchor so the space reads as deliberate. Highest impact: this is the product.

2. Progress page shows 0 READINESS / 0% COVERAGE / 0 ELEMENTS directly beside 2 EXAMS / 28 EXCHANGES, reading as lost data and breaking trust in the grading that justifies the subscription. Diagnose whether stats aggregation/rating-filter is wrong or the cards render before data loads (add skeletons); then show real values or an intentional 'Complete an exam to unlock your readiness' state instead of hollow zeros.

3. Cookie-consent bar overlaps the footer and, on mobile, makes Privacy/Terms/Accessibility links un-clickable — a legal/accessibility-compliance issue plus an 'unfinished' signal on the highest-traffic pages. Float the bar cleanly above the footer or reserve equivalent footer padding while it's visible; small fix, large credibility payoff.

4. Pervasive low contrast and 1px focus rings fail WCAG 2.2 AA (muted body text, 10px footer disclaimer, placeholders, small cyan/green labels, outline buttons, focus visibility) on a product that advertises an Accessibility link — both compliance risk and daily friction for older pilots studying in bright rooms. Fix at the token level: lift muted text toward ~0.7 alpha white, enforce a 12px floor for secondary text, and move to ring-2/3.

5. In-exam orientation and voice state are easy to lose: the session header (task, Q&A count, PAUSE/GRADE) scrolls off, voice status is split from the mic control, barge-in is undiscoverable, and there's no 'examiner is speaking vs. waiting' cue — eroding confidence in the core voice loop. Make the header sticky, consolidate mic+status, and add an explicit speaking/listening/ready indicator.

## Quick wins (high value, low effort)

- Float the cookie-consent bar above the footer (or add footer padding while visible) so Privacy/Terms/Accessibility stay clickable on mobile — pure CSS, removes a compliance + 'broken' signal everywhere.
- Constrain the exam chat container with a max-height and bottom-anchor the messages (flex justify-end / reverse layout) — collapses the void without redesigning the screen; biggest perceived-quality jump per line of code.
- Cap examiner/student message bubbles to ~65ch (and on mobile shrink or move the 56px avatars) so long ACS questions stop reflowing to one word per line and become readable under exam pressure.
- Make the session header position: sticky top-0 within the chat container with a subtle border/backdrop so task name, Q&A count, PAUSE and GRADE EXAM stay visible while scrolling.
- Replace hollow Progress zeros and empty Weak-Areas/Study-Plan boxes with intentional copy ('Complete an exam to unlock your readiness score') and add skeleton loaders so the page never looks broken mid-load.
- Bump every secondary/footer text below 12px up to a 12px floor and raise --color-c-muted toward rgba(201,209,217,0.7); upgrade focus rings from ring-1 to ring-2/3 — small token edits that fix a cluster of WCAG failures at once.
- Convert the worst all-caps offenders (long examiner/answer body text, feature-card descriptions, section subheads) to sentence-case IBM Plex Sans with leading-relaxed, keeping caps+glow for top-level headings, micro-labels, and badges — halves reading fatigue while keeping the cockpit look.
- Reframe the '1 Issue' badge into an inline, plain-language status near the input ('Answer saved — scoring pending / Some reference images unavailable') and give status badges/the pill a 44px min touch target.
- Raise the scanline opacity to ~0.03-0.04 (or remove it) and standardize the glow to CTAs + top-level headings + status only, so the theme reads as deliberate instead of half-applied — low effort, noticeable lift in 'premium' feel.
- Add a one-line session preview to the Discard/Resume flows ('Discard Preflight Preparation exam (1 Q&A)?') so users aren't relying on memory — the discard modal is already well-built, this just completes it.

## Full findings by severity


### BLOCKER

**Chat container is 75% empty void—profound spatial and trust issue** — `exam-entry-modal-desktop, exam-answered-desktop`  
Message container is massive (550px+ tall on desktop), but single examiner message floats at top, leaving vast blank gray area. This creates: (1) visual confusion—is the message there?; (2) sense of emptiness contradicting 'expensive tool' narrative; (3) poor spatial hierarchy (white space dominates content); (4) on mobile, blank scrollable area feels broken. Paying user's first impression: 'Is this working correctly?' DPE authority is undercut by an empty cockpit.  
→ *Fix:* Vertically center chat content OR implement reverse-scroll (new messages push up from bottom). Better: increase message spacing, boost scanline opacity (0.03–0.05), or place avatars/roles more prominently to fill visual weight. Add 'ping' animation (cyan pulse) for new messages; fade older messages slightly to draw focus downward. This fills void, adds motion, reinforces 'conversation in progress.'  _( Visual Hierarchy & Spatial Composition )_

**Massive empty void in chat container breaks conversation flow perception** — `exam-entry-modal-desktop.png, exam-answered-desktop.png, exam-question-desktop.png`  
The chat container has max-width: 2xl (42rem) with flex-1 (100% of remaining viewport height after header/input), but a single examiner message is pinned at the very top, leaving ~75% of the visible viewport area blank below it. This creates a visual impression that the app is broken, unfinished, or that content failed to load. In an oral exam context, where the examiner's question should dominate attention, the massive negative space undermines credibility and makes the interface feel hollow. The user cannot scan their conversation history because there isn't one yet, yet the abundance of empty space suggests the app is waiting or stalled.  
→ *Fix:* Reposition the chat container to be vertically centered OR anchor the first examiner message in the upper-middle area (top 20-30% of container) with intentional padding, making empty space feel deliberate rather than accidental. Alternatively: shrink the chat container's max-height to roughly 60-70% of viewport and position it higher, with the input area below. For voice-first exams, the message zone should feel 'present' even with one message—try a 3-column layout: left sidebar (session meta), center chat (narrower, ~32rem max), right sidebar (timing/progress)—OR simply move the message up via flexbox alignment and add visual anchoring (e.g., a subtle divider or 'Question' label above the first examiner message).  _( Fitts' Law / Visual weight distribution / Gestalt closure pr )_

**Live exam chat container is vastly oversized with single message orphaned at top** — `live-exam-desktop`  
The chat message container spans the full height between session header and input bar, but contains only 1-2 messages. This creates a massive ~75% empty void below the examiner's opening question. The examiner message floats at the very top with no visual anchoring, making it appear stranded and incomplete. A user pausing a session or re-reading the question has to scroll up significantly or perceive the space as a broken layout.  
→ *Fix:* Define a max-height constraint on the chat container (e.g., `max-h-96` or `h-[600px]`) and use `overflow-y-auto` to enable scrolling only when needed. Alternatively, use `flex-1` on the container but wrap messages in a flex-column with `justify-start` to pin content to the top, then let the input bar stay fixed at bottom. This ensures the examiner's message appears immediately below the header with natural spacing, and the empty space is only revealed when multiple messages accumulate.

**Chat container massive empty void breaks visual communication hierarchy** — `live-exam-desktop`  
The chat container on exam-answered-desktop.png is visually enormous (~75% of viewport) yet contains only a single examiner message at the top, leaving a massive empty void below. This breaks the perceptual scale of the interface: the eye reads size as importance, so an enormous empty box suggests the interaction is broken or waiting, when in fact the user should be focusing on the input area. On mobile (exam-question-mobile.png), this is even more jarring—the single line of text dwarfs the viewport.  
→ *Fix:* Either (1) Reduce chat container height to fit 3-4 visible messages + scrollable space (max 40-50% of viewport), rebalancing to give input area visual weight; or (2) if messages are sparse early on, replace the empty space with contextual guidance: "Your answer and the examiner's feedback will appear here" + visual prompt (e.g. 'speak your answer above ↑'). Consider a collapsible history panel for past messages, with the current Q&A pinned at bottom.

**Progress screen stat cards show '0' despite active sessions—credibility breach** — `progress-mobile`  
Stat cards display: 'READINESS: 0', 'COVERAGE: 0%', 'ELEMENTS: 0' despite header showing '2 EXAMS / 28 EXCHANGES'. Fundamental credibility breach: interface looks broken/uninitialized. Returning user with 2 completed exams expects non-zero readiness and coverage. Empty stats signal 'my data is lost' or 'app not tracking me.' Critical for paid app—users distrust calculations.  
→ *Fix:* Verify /api/session?action=stats endpoint returns correct aggregates for completed sessions. Check: (1) Is filtering by selectedRating correct? (2) Is backend including all sessions, not just recent? (3) Are assessment scores ('satisfactory', 'partial', 'unsatisfactory') saved on completion? Once verified, display real values. If data missing, add fallback: 'Complete your first exam to see progress' with CTA to '/practice'.  _( Data Integrity & Trust )_

**Cookie consent banner occludes footer links on mobile** — `progress-mobile, landing-mobile, home-mobile (all mobile layouts)`  
The cookie consent banner is fixed at bottom (z-50) with no bottom padding on the page. On mobile, the banner physically overlaps and hides footer Privacy Policy and other links, making them un-clickable. Users cannot access accessibility info or privacy without closing the banner first.  
→ *Fix:* Add pb-[60px] or pb-[80px] (padding-bottom) to the body/main content container when the banner is visible, or reposition the banner above the footer. Alternatively, render the banner as a modal overlay (backdrop) that doesn't push layout, so it floats above content instead of covering it.


### HIGH

**Error badge ('1 Issue') has insufficient touch size and unclear contrast** — `exam-answered-desktop, exam-answered-mobile`  
The red '1 Issue' badge (bottom-left) is approximately 40–50px wide but uses a very small font (text-[11px] or smaller). The badge is < 44x44px, violating WCAG 2.5.5 (target size). Additionally, if the text color is not white on the red background, contrast may be below 4.5:1. The badge is a critical error indicator that should be highly visible and tappable.  
→ *Fix:* Expand the badge to at least 44x44px (or min-h-11 min-w-44 in Tailwind). Use white text on the red background (white on #ff3b30 yields ~5.5:1, passing AA). Optionally add an icon or expand the badge to show '1 Issue Found' with more readable text size.

**Message bubbles not vertically centered in oversized avatar regions** — `exam-answered-desktop.png, exam-question-desktop.png`  
Message bubbles render at the top of their flex container, while 14x14 avatars sit below them (mt-1). For short messages (1-2 lines), this creates visual misalignment: the avatar dangles at the bottom of the bubble instead of aligning with the bubble's centroid. In award-caliber messaging UIs (iMessage, Slack, Discord), the avatar is vertically centered to the message. The current layout breaks this baseline pattern and looks amateurish.  
→ *Fix:* Change the avatar alignment from mt-1 (margin-top-1) to items-center on the parent flex container, removing hardcoded mt-1. If avatars must be pinned to the bottom of the message, make that intentional with mb-0 and document it. Better: use items-start gap-3 and apply alignment to the bubble itself (align-self: center or flex-grow to center the bubble within the flex-direction: row group).  _( Baseline alignment / Spatial consistency )_

**Mobile layout crushes the 3-column message design into a 2-column layout without clear rethinking** — `exam-answered-mobile.png, exam-question-mobile.png`  
On mobile (390px width), the message bubble + avatar design scales poorly. The examiner message is narrower (due to max-w-[calc(100%-4.5rem)] accounting for avatars), and text reflows into single words per line. The avatars (w-14 h-14) still take up 25% of the width, leaving only ~75px for text. Reading a long examiner question becomes tedious. The 'gap-3' between avatar and bubble is proportionally huge on small screens.  
→ *Fix:* On mobile (<640px), hide avatars entirely (or render them as tiny icons in the message header instead of alongside). Expand the message bubble to near-full width (max-w-[calc(100%-1rem)]). The flex-col should change to stack avatar (if shown) above the bubble, not beside it. This is a standard mobile messaging pattern (e.g., iMessage on iPhone).  _( Responsive design / Touch-friendly text sizing )_

**Placeholder text contrast below 3:1 on input fields** — `exam-entry-modal-desktop, exam-answered-desktop, exam-question-mobile`  
The exam input placeholder 'Type your answer... (Enter to send)' uses dim gray text on a dark bezel background. Placeholder text measured at likely text-c-dim (#7d8898) or similar yields <3:1 contrast, failing WCAG 4.5.2. Placeholder contrast is not required to be 4.5:1, only 3:1, but current implementation likely falls short.  
→ *Fix:* Increase placeholder text color to at least text-c-muted (#8b949e) or lighter. Test with a color picker to ensure 3:1 minimum contrast. Or, add a label above the input 'Your Answer:' in solid text-c-text to compensate for weak placeholder visibility.

**No visual affordance for typing state during TTS playback** — `exam-entry-modal-desktop.png (after first examiner message)`  
When voice is ON and the examiner's message is being streamed + spoken, the UI does NOT distinguish between (a) waiting for examiner response (loading indicator visible, input disabled), (b) DPE speaking (TTS active, user cannot interrupt clearly), and (c) ready for user input. The 'mic' button changes color (red = listening), but the examiner message area has no visual cue that TTS is in-progress. For users with hearing aids or in loud environments, this is a critical UX gap: they cannot tell if the DPE is still 'talking' or waiting.  
→ *Fix:* Add a subtle animated border/glow to the examiner message bubble during TTS playback (e.g., a 2-3px dashed border in c-amber that fades in/out). Alternatively, render a live waveform or playing-indicator icon in the corner of the message. The mic button's text should also change: 'INTERRUPT' (not just icon shift) when TTS is active. This provides sighted and non-sighted users with clear feedback.  _( Affordance visibility / Voice UI feedback / WCAG 2.1 Level A )_

**Empty states lack direction; unclear what actions users should take next** — `home-desktop, progress-mobile`  
On home-desktop.png, the HOME greeting is polished but the flow from greeting → quick action buttons → Pro Tips is linear and static. For new users, there's no guided onboarding visible in the HOME page—they must infer that 'START YOUR FIRST EXAM' leads to a config form. On progress-mobile.png, the empty state ('NO PROGRESS DATA YET') has a CTA but no explanation of what a 'practice exam' entails. The landing page has a 'THREE STEPS' section, but the dashboard doesn't reinforce those steps, creating a discontinuity.  
→ *Fix:* Add a progressive onboarding banner on the HOME page for users with zero sessions: '1. Configure your exam. 2. Answer the examiner's questions. 3. Review your score & weak areas.' Link to /practice with a pre-populated config step (or show config inline). For progress empty state, add a 1-2 sentence explanation: 'Your first exam will benchmark your knowledge across FAA ACS tasks. You'll practice orally (or by typing) and see your strengths and gaps.'

**Excessive monospace uppercase creates cognitive fatigue** — `landing-desktop, exam-answered-desktop, home-desktop`  
Nearly every headline and label uses ALL-CAPS monospace (JetBrains Mono). Landing reads: 'YOUR DPE IS READY', 'THREE STEPS TO CHECKRIDE CONFIDENCE', 'SEE IT IN ACTION', 'BUILT FOR SERIOUS CHECKRIDE PREP'. While thematic to avionics, relentless uppercase creates eye fatigue and reduces perceived sophistication. Premium SaaS (Apple, Stripe) limit caps to headlines or accents, not entire label systems. This reads as 'gaming' or 'retro-pastiche' rather than 'professional aviation prep tool.'  
→ *Fix:* Introduce sentence/title case for body labels and non-critical labels. Retain ALL-CAPS only for: top-level section headings, micro-labels in bezel headers (e.g., 'DPE EXAMINER'), and status badges. Change 'FAA REFERENCES' label to 'FAA References' in normal case. This halves cognitive load while preserving the cockpit aesthetic.  _( Typographic Hierarchy & Readability )_

**Focus indicators are too thin and potentially invisible** — `landing-desktop, home-desktop, home-mobile, and all interactive elements`  
Buttons and links use `focus:ring-1 focus:ring-c-amber`, a 1px ring. Standard accessibility guidance recommends 2–3px focus rings (especially on dark backgrounds). At 1px with glow effects, the focus ring is difficult to see for users navigating by keyboard. Many custom components lack visible focus states entirely.  
→ *Fix:* Upgrade all focus rings to `focus:ring-2` or `focus:ring-[3px] focus:ring-c-amber` with `focus:ring-offset-0` or `focus:ring-offset-1` to create separation from the element. Add a prefers-reduced-motion media query to remove motion but keep the ring visible. Test keyboard navigation (Tab key) to ensure every button/link shows a clear focus indicator.

**Footer overlapped by cookie consent bar on multiple screens** — `landing-desktop, home-desktop, progress-mobile`  
The cookie consent banner is `fixed bottom-0 z-50` without accounting for footer height. On the landing, home, and progress pages, the footer content is partially or fully obscured by the consent bar, making footer links (Help, Privacy, Terms, Accessibility) unclickable or invisible. Users on mobile especially cannot dismiss the banner or access footer functionality.  
→ *Fix:* Restructure to: (1) Move the cookie consent into the footer component and use relative positioning, or (2) give the footer a `pb-[consent-height]` conditional padding when consent is visible, or (3) if the banner must be fixed, add `bottom-[cookie-height]` to the footer so it doesn't overlap. Test footer visibility at 390px width to ensure all links remain accessible.

**Excessive uppercase and monospace density collapses messaging hierarchy** — `landing-desktop, landing-mobile, home-desktop, progress-mobile`  
Almost every label, heading, and section uses uppercase monospace simultaneously: 'THREE STEPS TO CHECKRIDE CONFIDENCE', 'BUILT FOR SERIOUS CHECKRIDE PREP', 'SEE IT IN ACTION', 'PROGRESS', 'WEAK AREAS', 'STUDY PLAN'. When everything is uppercase + monospace + glow, nothing stands out. Users cannot quickly scan for primary vs. secondary messages. The cockpit aesthetic is consistent but sacrifices scannable hierarchy.  
→ *Fix:* Introduce hierarchy tiers: (1) Primary headings (h1/h2) use uppercase monospace + glow, but at larger size (3xl–5xl). (2) Section subheadings use *sentence case* with normal weight, no glow, slightly smaller. (3) Labels/tabs/badges use uppercase monospace but *smaller* (xs–sm) with lower opacity. Example: 'THREE STEPS TO CHECKRIDE CONFIDENCE' (current) → keep as h2 at 2xl; 'PROCEDURE' (current comment, xs) → make that size, no caps; 'SET YOUR RATING' (h3 in cards, current sm) → drop to xs, remove glow. This preserves the cockpit feel while creating visual rest stops.

**Footer disclaimer text fails contrast at 10px font size** — `landing-desktop, pricing-desktop, home-desktop, progress-desktop, and all footer contexts`  
Footer disclaimer uses text-[10px] text-c-muted (#8b949e on #080c12), yielding ~4.8:1 contrast. At sizes below 18.66px, WCAG AA requires 4.5:1 minimum; this is at the threshold and fails for users with low vision or older screens. The same text-c-muted class is used throughout secondary UI at small sizes.  
→ *Fix:* Increase footer disclaimer to text-xs (12px) minimum and upgrade color to text-c-text (#c9d1d9) for ~6.5:1 contrast, or reduce font size only if contrast is boosted to text-c-text. Apply the same principle to all secondary labels < 18.66px: ensure 4.5:1 minimum contrast.

**Color saturation and contrast vary widely across screens** — `landing-desktop, progress-mobile, pricing-desktop`  
Brand palette (amber #f5a623, cyan #00d4ff, green #00ff41) vibrant on #080c12 base, but usage uneven: landing = amber-heavy, exam = cyan-heavy, progress = multi-color. Creates 'visual instability'—each screen feels different in mood/intensity. Contrast occasionally too low: muted text (#8b949e) on dark panels (#0d1117) = 4.3:1 (barely WCAG AA), feels dim. Overall effect: 'visually exhausting' not 'premium and trustworthy.'  
→ *Fix:* Establish strict color hierarchy per screen: (1) Landing = 80% amber, 15% cyan, 5% green. (2) Exam/Chat = 70% cyan, 25% amber, 5% green. (3) Progress = 60% cyan, 30% amber, 10% green. Enforce contrast minimums: upgrade muted text on dark backgrounds to rgba(201, 209, 217, 0.65) for 6.5:1 contrast. Test all text at 14px and below on actual monitors.  _( Color Harmony & Contrast (Accessibility) )_

**Sticky session header can scroll off-screen, losing task context** — `live-exam-desktop`  
The session header (task name, 'N Q&A', VOICE, PAUSE, GRADE EXAM buttons) is positioned in the content flow and scrolls away when the chat grows. During a long exam, the user loses the current task label and button bar, forcing them to scroll back up to reference what they're being tested on or to pause the session.  
→ *Fix:* Make the session header `sticky top-0` (or use `position: sticky; top: 0;` in CSS) with a `z-index` lower than modals. Add a subtle shadow or border to visually separate it from scrolling content. Ensure it stays within the scrollable chat container boundaries, not the entire viewport, so the header scrolls with the chat but remains visible within it.

**Session header can scroll off-screen, breaking context awareness mid-exam** — `live-exam-desktop`  
The session header (task name, Q&A count, VOICE indicator, PAUSE, GRADE EXAM buttons) is positioned above the chat container without sticky positioning. On longer exams or when the chat scrolls, this header disappears, leaving the user unaware of: (1) which task they're on, (2) how many exchanges have occurred, (3) whether voice is active. This is a critical context loss for a task-focused interface. Returning users resuming an exam also lose orientation.  
→ *Fix:* Make the session header sticky (position: sticky; top: 0) with a subtle background (slightly elevated from page bg) so it remains visible during chat scroll. Reduce header height if needed (task name + count on one line; controls on second) to minimize space impact. Add a visual breadcrumb or progress indicator (e.g. 'Exchange 3 of ~10') to reinforce where the user is in the session.

**Progress page displays 0 readiness/coverage despite active history; returns user confusion** — `progress-desktop`  
On progress-desktop.png, the stat cards show '0 READINESS / 0% COVERAGE / 0 ELEMENTS' even though the header states '2 EXAMS / 28 EXCHANGES' and RECENT EXAMS shows two sessions with INCOMPLETE badges. This is a **data-consistency error or UI bug**: either the stats aggregation is broken, the component is initializing before data loads, or the component is filtering by a class/rating the user hasn't interacted with. Returning users see a 'broken' experience that undermines confidence in the app's accuracy.  
→ *Fix:* Investigate stat aggregation logic: verify that (1) lifetime stats endpoint sums across all sessions+ratings, not just the filtered session; (2) UI waits for data before rendering stats (show skeleton loaders for each stat card while loading); (3) if user has no element scores in the selected class, show a targeted empty state: 'No data yet for ASEL. Complete an exam to see coverage.' Ensure RECENT EXAMS list is always consistent with the stat cards above.

**Progress page stat cards show '0 READINESS / 0% COVERAGE / 0 ELEMENTS' despite active sessions** — `progress-mobile`  
The layout shows stat cards reading '0 READINESS', '0%' coverage, and '0 ELEMENTS' even though the session list shows '2 EXAMS' and '28 EXCHANGES'. This is a spacing/content visibility issue where the calculated stats (e.g., `coveragePct`, `ratingSessions`) are not rendering or are returning stale values. The empty state design makes the page feel broken to a returning user.  
→ *Fix:* Debug the stats calculation in `/api/session?action=stats` to ensure lifetime aggregates are computed correctly and include all sessions, not just a recent 20-row window. Verify that the skeleton loader appears while stats load (currently shows 'LOADING TREEMAP' for chart but not the summary cards). Add a fallback message like 'Complete a practice session to see your progress' if stats are legitimately empty, rather than showing zero values.


### MEDIUM

**All-caps monospace small type reduces readability and excludes dyslexic users** — `all screens (pervasive)`  
The entire UI uses `font-mono uppercase text-xs text-[11px] text-[10px]` throughout. All-caps monospace is 20–35% slower to read and triggers eye strain. Dyslexic users often struggle with monospace and all-caps (which removes ascenders/descenders, key visual markers). Tiny monospace (10–12px) compounds the problem and is barely readable on standard monitors.  
→ *Fix:* For body text and secondary labels, switch to `font-sans` (IBM Plex Sans is already in the stack). Reserve monospace for code snippets, technical references, and headings. For labels that must stay monospace (DPE brand identity), increase minimum font size to 13–14px and consider sentence case for readability (e.g., 'Start Practicing Free' instead of 'START PRACTICING FREE'). Audit for critical information in all-caps monospace at < 12px and convert.

**Text-shadow glow effects blur character edges and reduce contrast perception** — `all screens with glow-a, glow-g, glow-c classes`  
The `.glow-a`, `.glow-g`, `.glow-c` text-shadows use `color-mix()` with 40% opacity to create a CRT glow effect. While visually distinctive, these shadows soften character edges, reducing perceived contrast and making small text harder to read. Users with low vision or astigmatism are especially affected. WCAG does not credit decorative effects; they unintentionally hurt accessibility.  
→ *Fix:* Reduce glow opacity to 15–20% or apply only to headings (h1, h2) and large text (> 24px). For body text and labels, remove glow entirely or use a subtle, tighter shadow (e.g., `0 0 4px rgba(..., 0.1)`). Test with screen reader users and those with vision impairments to measure real-world readability impact.

**'Necessary Only' button contrast insufficient without fill** — `cookie consent banner and customize modal (all pages)`
