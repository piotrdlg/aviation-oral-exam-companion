01Principles

The engine is built around one separation: the server decides what must be covered and how it is graded; the LLM decides only how to ask and — in scenario mode — what order feels natural. Every product promise (linear order, difficulty level, weak-area targeting, scenario realism) is enforced by deterministic server code, never by prompt hope.

Invariant Non-linearity changes the sequence of elements, never the set. The set — which ACS elements this session covers, at which difficulty, with what question budget — is frozen into the exam plan at session start and is identical across all modes.

The flag controls availability and experiment state, never a running session:

Guarantee A student who picks “Area by Area” gets the exact ACS document order at their chosen difficulty regardless of any flag state — this is asserted by automated tests that run the full API route with the flag forced on and verify byte-identical linear behavior.

05Difficulty: the end-to-end guarantee

Difficulty is enforced at three independent points, and all three are upstream of (or shared with) the scenario layer — which is why no mode can break it:

Under mixed, each element is served at its ACS-derived default (difficulty_default); under a fixed level, the set was already filtered, and the contract pins both questioning depth and grading strictness to that level. The scenario engine's advanceToElement calls the identical contract builder as the linear path — verified by shared-code construction and by tests.

06Session lifecycle

Advancement rule (identical in every mode)

  • satisfactory → advance to the next element.
  • partial / unsatisfactory → one follow-up on the same element (budget: follow_up_max_per_element = 1), then advance regardless.
  • ungraded (infrastructure failure, not a judgment) → stay; the element deserves a real attempt.
  • unsatisfactory additionally arms a bonus question (max 2 per session) so a stumble doesn't shrink the exam.

07The Scenario Engine

Scenario mode replicates how a real DPE runs an oral: assign a flight, then let weather, performance, airspace and ADM questions emerge from the scenario's own logic. Three components:

1 · The spine (once per session)

One Claude call at exam start produces a structured scenario — aircraft, mission, conditions, pilot role, human pressures — plus hooks (scenario details that naturally open specific planned elements) and 1–2 events (mid-exam developments like “the alternator light flickers”). It is validated hard (hooks must reference planned elements; strays dropped), retried once, and falls back to one of nine hand-written scenarios per rating. Generation runs after the first question is already on its way — exam start never waits.

2 · The adjacency table (offline)

element_adjacency holds the top-12 related elements for every K/R element (14,599 rows), blended from semantic similarity (0.60), FAA-source co-occurrence (0.25) and ACS structure (0.15) — every score carries its explainable breakdown for CFI audit.

3 · The transition policy (each transition)

Proof, not promises Gate 1 (offline, 40 blind-judged pairs): scenario beat linear 90.0% overall [76.9–96.0], 90.9% on cross-topic probing, with accuracy and citation grounding non-inferior and all cost budgets met. Gate 2 (production A/B inside “Across ACS” sessions) ships with its own report tooling and rollout runbook; the rollout decision is the owner's.

08Grading policy

Per answer

Every student answer gets a structured assessment from a dedicated grading call (separate from the examiner): a score, the primary ACS element addressed, other elements genuinely touched, misconceptions, and a source summary citing FAA references.

ScoreMeaningPointsEffect
satisfactoryMeets the ACS standard at this difficulty1.0advance; mentioned elements may earn coverage credit
partialCore present, a material piece missing0.7one follow-up, then advance
unsatisfactoryCore wrong, absent, or dangerously misapplied0.0follow-up + arms a bonus question (max 2)
ungradedInfrastructure failure — not a judgmentexcludedstay on the element; never counts against the student
Anti-gaming rule “Credited by mention” — elements you covered thoroughly while answering something else — is granted only on satisfactory answers, and a hallucinated element code from the model can never poison the record: every attributed code is validated against the task's real element list.

Per exam (V2, plan-based)

The final grade is computed from the plan, mirroring the FAA's 70% oral standard:

  • Score = points earned ÷ elements in the plan (partial = 0.7). Pass ≥ 70%.
  • Per-area gating: any area with ≥2 attempts must individually reach 60% — a strong overall score cannot hide a collapsed area.
  • Critical areas (PA/CA: Area I; IR: Areas I & III) are flagged if they contain any unsatisfactory answer.
  • Incomplete only when the session genuinely ended early — a naturally finished exam always grades pass/fail.

Identical inputs in every mode: scenario transitions change which question comes next, never how the answer is scored or counted.

09Progress records — what is written, when

Write-ordering invariant

Per exchange the chain is: student transcript → assessment attached → element_attempts → citations. The examiner's reply persists independently, so a grading-service failure can never lose conversation history, and an ungraded marker is recorded instead of a fake grade.
