00The 5 things you need to know first

1. "Ground school" is already gone. What you remember as the free/"ground school" tier is stored in the database as checkride_prep today. The value ground_school was a launch-era name that was migrated away on 2026‑02‑18 (20260218000001_simplify_voice_tiers.sql) and no live user has it. New signups default to checkride_prep (auth/callback/route.ts:64). So your rename is really checkride_prep → trial, not ground_school → trial.
2. There are 2 real tiers + 1 "override", not 3 tiers. The stored tier column has exactly two live values: checkride_prep (free) and dpe_live (paid). Your "tester / free until revoked" is not a tier today — it's a separate row in user_entitlement_overrides (key paid_equivalent) that simulates paid. Making "tester" a first-class tier is a genuine new concept, not a rename.
3. The "7‑day trial" is two unrelated things, and neither matches your description.
  • Free tier cap = 3 exams, count-only, forever — there is no 7‑day window enforced on the free tier today (session/route.ts:76 counts all exams ever; the FREE_TRIAL_EXPIRY_DAYS=7 constant only stamps a per‑session field that isn't gated on).
  • Stripe 7‑day trial = on the paid plan, and it requires a credit card (Stripe's default payment_method_collection:'always'; the route never overrides it).
Your desired "trial = 7 days AND 3 exams, no card, then pay" merges these into one app-enforced trial and removes the Stripe trial. That's new behaviour, covered in §6.
4. trialing is Stripe's word, not ours. The subscription_status values (trialing, active, past_due, …) arrive verbatim in Stripe webhooks. You cannot rename them. The "trial" concept you want to name lives in our tier column, separate from Stripe's status.
5. Tiers are stored values that gate access — so a rename is a data migration, not a find-replace. tier === 'dpe_live' is the load-bearing check that unlocks unlimited exams (5+ places). The strings live in the production user_profiles.tier column, in system_config keys, in DB CHECK constraints, and in the signup trigger. Code and database must change together or paid users lose access mid-deploy (full detail in §8).

03The lifecycle, step by step

1. Sign up ──▶ auth/callback creates profile: tier=checkride_prep, status=none, has_trialed=false
2. Practice free ──▶ session/route counts non-onboarding exams; allows while count < 3 (no time limit)
3. Start exam #4 ──▶ 403 trial_limit_reached ──▶ practice page shows paywall modal → /pricing
4. Click a plan ──▶ stripe/checkout: if !has_trialed adds trial_period_days:7; card is collected (default)
5. Checkout done ──▶ webhook checkout.session.completed: tier→dpe_live, status→trialing, has_trialed→true
6. Day 7 ──▶ Stripe charges the card: status→active, tier stays dpe_live
7. Payment fails ──▶ status→past_due, tier kept ~2 wks (dunning grace), then subscription.deleted → tier→checkride_prep
8. Cancel ──▶ subscription.deleted: tier→checkride_prep; has_trialed preserved (no second trial ever)

Where each write happens: signup default auth/callback/route.ts:64 · quota gate api/session/route.ts:38,76 · checkout api/stripe/checkout/route.ts:75 · all tier writes api/stripe/webhook/route.ts (checkout.session.completed, customer.subscription.updated/deleted, invoice.paid, invoice.payment_failed, charge.refunded) · fast-path grant api/stripe/status/route.ts:47.

The unlimited gate everywhere is the same line: tier === 'dpe_live' → isPaid/isPaying (api/exam/route.ts:420, api/session/route.ts:38, lib/kill-switch.ts:39, pricing/page.tsx:127).

05"Free until revoked" today

Two paths grant free access without paying — neither is a tier:

  1. Comp override (your "tester"). An admin POSTs api/admin/user-overrides → upserts user_entitlement_overrides with paid_equivalent, active=true, optional expires_at, a reason. getUserTier() then returns dpe_live for that user. Revoke = set active=false. This is exactly "free until revoked", just implemented as an override layer rather than a tier value.
  2. Instructor courtesy. An approved instructor with a connected paid student auto-resolves to checkride_prep courtesy (lib/instructor-entitlements.ts).
Design choice for you: "tester" can stay an override (zero schema change, already revocable, already admin-managed) or become a first-class tier='tester' value. The override path is dramatically lower-risk and already does what you described. A real tier only helps if you need testers to behave differently from paid (e.g., different quotas or analytics bucket).

09Recommendation

You can get the clarity you want with far less risk by separating what users see from what's stored:

  1. Rename the words, not the stored values (do now, ~zero risk). Introduce one constants module — e.g. TIER_LABEL = { checkride_prep:'Trial', dpe_live:'Paid' } — and route every admin/UI/email label through it. The database keeps checkride_prep/dpe_live; humans see "Trial"/"Paid". This kills the confusion immediately with no migration and no gate risk.
  2. Keep "tester" as the override (recommended). It's already "free until revoked" and admin-managed. Just surface it in the admin UI as a "Tester" badge. Only promote it to a real tier='tester' if testers must differ from paid in quotas/analytics.
  3. Do the behaviour change separately (the real value). If you want the genuine "7 days + 3 exams, no card, then pay" model: add the 7‑day window check in api/session and drop trial_period_days from api/stripe/checkout. That's a small, testable change independent of any rename.
  4. Only do a full DB string rename if you truly need it. If so: one migration (UPDATE + CHECK + trigger + config keys) deployed atomically with the code (TypeScript union flip is your safety net), behind a staging dry-run, with the analytics old→new map prepared. Budget ~1–2 days + a careful release, not a find-replace.
Bottom line: the naming you want is a 1‑hour label change; the trial-behaviour you described is a small, worthwhile change; a full stored-value rename is the only expensive/risky piece — and it's optional. I'd do 1+3 now, keep tester as an override, and skip the stored-value rename unless there's a concrete reason.
